# driver_base 工作流程与基类优缺点分析

> 生成时间: 2026-04-22
> 分析范围: `d:\kuanghu-poc\driver_base`

## 1. 目录定位

`driver_base` 是一个面向 Playwright 异步自动化场景的基础框架层，目标不是直接承载某个具体站点的业务逻辑，而是提供一套可复用的“多浏览器并发执行基类”。

它主要解决下面几类共性问题：

- 如何管理多个浏览器 worker
- 如何在单个浏览器内并发创建多个独立 context
- 如何让每个任务带上自己的 Cookie、代理和超时配置
- 如何把页面初始化、上下文初始化、任务处理逻辑暴露成钩子
- 如何在异常出现后做统一失败分类和浏览器回收

从职责上看，这个目录本质上分成三层：

1. 调度层: `MultiBrowserScraperBase`
2. 执行层: `BrowserWorker`
3. 数据模型层: `WorkerContext`、`FailureCategory`、`TaskPolicy`、`CookieMode`

## 2. 目录结构与职责

### 2.1 文件清单

| 路径 | 作用 |
| --- | --- |
| `driver_base/__init__.py` | 统一导出公共类型，方便业务层直接导入 |
| `driver_base/multi_browser_scraper_base/__init__.py` | 主基类，负责 worker 池、任务调度、异常分类、回收协同 |
| `driver_base/browser_worker/__init__.py` | 单个 worker 的浏览器生命周期管理与任务执行 |
| `driver_base/worker_context/__init__.py` | 向业务钩子传递只读 worker 状态快照 |
| `driver_base/failure_category/__init__.py` | 失败分类枚举 |
| `driver_base/task_policy/__init__.py` | 任务策略数据结构，目前处于预留状态 |
| `driver_base/cookie_mode/__init__.py` | Cookie 使用模式枚举 |
| `driver_base/tests/` | 针对导入、失败分类、worker 回收与健康检查的回归测试 |

### 2.2 各模块关系

可以把整体关系理解成下面这张图：

```text
业务子类
  └─ 继承 MultiBrowserScraperBase
       ├─ run_tasks() 负责并发调度
       ├─ _acquire_worker() / _release_worker() 管理 worker 使用权
       ├─ _run_single_task() 包装单任务执行、异常分类、状态记录
       └─ BrowserWorker.run_task() 真正创建 context/page 并执行业务逻辑
            ├─ ensure_started() 启动或复用浏览器
            ├─ _resolve_context_options() 解析代理与上下文参数
            ├─ initialize_context() / initialize_page() 扩展钩子
            └─ process_task() 由业务子类实现
```

## 3. 核心设计思路

### 3.1 为什么要拆成 `MultiBrowserScraperBase` 和 `BrowserWorker`

这里的拆分很关键：

- `MultiBrowserScraperBase` 负责“谁去执行任务”
- `BrowserWorker` 负责“拿到任务后怎么执行”

也就是说：

- 调度层不直接关心浏览器到底怎么启动
- 执行层不直接关心整个池子的并发分配策略

这种拆分带来的好处是，业务层继承时只需要关注 `process_task()`，不用自己处理浏览器池、并发、回收这些基础设施。

### 3.2 为什么使用 Browser + Context 的结构

当前实现不是“每个任务启动一个浏览器”，而是：

- 一个 worker 持有一个浏览器实例
- 一个浏览器实例内部可以并发创建多个 context
- 每个任务在自己的 context 和 page 里执行

这意味着：

- 浏览器启动成本被摊薄
- 多任务之间通过 context 进行隔离
- 每个任务仍然可以使用独立 Cookie 和代理

这是自动化框架里比较常见、也比较实用的一种折中方案。

## 4. 实际工作流程

下面按真实调用顺序说明这个目录的运行方式。

### 4.1 初始化阶段

业务方通常会先继承 `MultiBrowserScraperBase`，只实现自己的 `process_task()`，例如：

```python
class MyScraper(MultiBrowserScraperBase):
    async def process_task(self, page, task_data, worker):
        await page.goto(task_data["url"])
        return {"ok": True}
```

随后创建实例时，会注入全局配置，例如：

- `browser_pool_size`: 启动多少个浏览器 worker
- `max_contexts_per_browser`: 单个浏览器允许多少个并发 context
- `task_timeout_ms`: 单个任务总超时
- `default_cookies`: 默认 Cookie
- `default_proxy`: 默认代理
- `recycle_browser_after_tasks`: 执行多少任务后请求回收浏览器
- `recycle_browser_after_failures`: 连续失败多少次后请求回收浏览器

这些配置先保存在 `MultiBrowserScraperBase` 上，并不会立即启动浏览器。

### 4.2 启动阶段

当调用 `start()` 或 `run_tasks()` 时，基类才会真正初始化 worker 池：

1. 创建 `browser_pool_size` 个 `BrowserWorker`
2. 给每个 worker 注入一组回调
3. 初始化 `_worker_active_counts`
4. 设置 `_started = True`

这里注入给 `BrowserWorker` 的不是固定配置对象，而是一组“工厂函数/钩子”，包括：

- `build_launch_options`
- `build_context_options`
- `_build_cookie_payload`
- `initialize_context`
- `initialize_page`
- `process_task`

这使得 `BrowserWorker` 变成了一个通用执行器，不必知道上层具体业务。

### 4.3 任务入口阶段

业务层调用 `run_tasks(tasks)` 后，会发生两件事：

1. 先确保 worker 池已启动
2. 使用 `asyncio.gather(..., return_exceptions=True)` 并发执行所有任务

这说明 `run_tasks()` 的设计目标是“批量并发执行”，并且它会保留每个任务的异常对象，而不是在第一个异常出现时直接中断全局。

### 4.4 单任务调度阶段

每个任务都会进入 `_run_single_task()`，主要步骤如下：

1. `normalize_task(task_data)` 先做任务标准化
2. `_acquire_worker()` 从池中选择一个可用 worker
3. `_build_worker_context()` 构建当前 worker 的只读快照
4. `_run_worker_task()` 调用 worker 执行任务
5. 成功时调用 `worker.record_success()`
6. 失败时调用 `classify_failure()` 做异常分类
7. 失败分类结果再交给 `worker.record_failure()`
8. 最终在 `finally` 中调用 `_release_worker()`

这段流程说明：调度层不仅负责分配 worker，也负责把任务成功、失败、释放三个阶段串成完整生命周期。

### 4.5 worker 获取策略

`_acquire_worker()` 的挑选规则不是随机的，而是按以下优先级选最小值：

1. 当前活跃任务数更少
2. 连续失败次数更少
3. 距离上次回收已执行任务数更少
4. `worker_id` 更小

同时，它会过滤掉两类 worker：

- 已达到 `max_contexts` 上限的 worker
- 正处于回收中的 worker

这套策略的含义是：

- 尽量均衡负载
- 尽量避开状态不稳定的 worker
- 尽量避免刚跑了很多任务、接近回收阈值的 worker 被继续打满

### 4.6 worker 执行阶段

当任务进入 `BrowserWorker.run_task()` 后，执行顺序如下：

1. `ensure_started()` 确保底层浏览器已启动且健康
2. `_create_cookies_payload(task_data)` 生成任务 Cookie
3. `_resolve_context_options(proxy)` 解析本次任务的 context 配置
4. `browser.new_context(...)` 创建独立浏览器上下文
5. 调用 Scrapling session 的内部 context 初始化逻辑
6. 调用基类钩子 `initialize_context(...)`
7. `context.new_page()` 创建页面
8. 设置默认超时
9. 应用 stealth
10. 调用基类钩子 `initialize_page(...)`
11. 调用业务侧 `process_task(...)`
12. 在 `finally` 中关闭 page 和 context

也就是说，每个任务都是“共享浏览器，但独占 context/page”的执行模型。

### 4.7 浏览器启动与复用阶段

`BrowserWorker.ensure_started()` 是整个执行层里最关键的方法之一，它承担三件事：

1. 判断当前浏览器是否已经存在且健康
2. 在并发场景下保证浏览器只会被启动一次
3. 启动后给 session 打一个支持任务级代理的补丁

内部关键点如下：

- 通过 `_start_lock` 避免多个协程同时重复启动浏览器
- 通过 `_is_browser_healthy()` 检查浏览器连接和版本可用性
- 如果旧的 session/browser 引用失效，会先 `close()` 再重建
- 启动完成后，给 `AsyncStealthySession` 动态挂载 `_build_context_with_proxy`

最后这一点说明当前实现为了支持“每个任务不同代理”，对第三方库做了运行时 patch。

### 4.8 Cookie 处理流程

Cookie 的处理分成两层：

1. `run_task()` 里通过 `_build_cookie_payload()` 取当前任务 Cookie
2. `_build_cookie_payload()` 再调用 `load_cookies()` / `normalize_cookies()` 做统一格式化

支持的输入形式有：

- `None`
- `name=value; name2=value2` 形式的字符串
- `dict[str, str]`
- `list[dict[str, Any]]`
- 指向 Cookie 文件的路径

优点是业务接入很方便，很多历史脚本可以直接复用原有 Cookie 数据。

### 4.9 代理处理流程

代理也支持两级来源：

1. 任务级 `task_data["proxy"]`
2. 实例级 `default_proxy`

最终由 `resolve_task_proxy()` 解析，然后交给：

- `BrowserWorker._resolve_context_options()`
- 或 session 上被 patch 的 `_build_context_with_proxy()`

这意味着代理粒度是“任务级”，而不是“整个 scraper 实例级”。

### 4.10 失败分类与回收流程

当任务抛出异常时，`_run_single_task()` 会调用 `classify_failure()` 进行统一归类。

当前主要分为五类：

- `PROXY_ERROR`
- `BAN_ERROR`
- `TASK_ERROR`
- `WORKER_ERROR`
- `UNKNOWN`

分类依据主要来自：

- 异常类型
- 异常消息关键字

分类之后，worker 会调用 `record_failure()` 更新：

- `tasks_since_recycle`
- `consecutive_failures`
- `request_recycle`

触发回收请求的条件主要有：

- 失败被判定为 `WORKER_ERROR`
- 累计任务数达到阈值
- 连续失败数达到阈值

注意这里是“请求回收”，不是立刻强制关闭。

### 4.11 安全释放与回收阶段

`_release_worker()` 的处理比较谨慎：

1. 先把该 worker 的活跃任务数减一
2. 如果活跃任务数已经为 0 且 `request_recycle=True`，则把它放入 `_workers_recycling`
3. 回收中的 worker 不允许再次被分配
4. 调用 `worker.recycle_if_needed(active_tasks=0)` 真正执行关闭
5. 回收结束后再移出 `_workers_recycling`

这个设计的重点是：只有在 worker 已空闲时才回收浏览器，避免中途打断仍在运行的任务。

## 5. 业务方如何接入

从 `flow/demo.py` 这类调用方式可以总结出一个典型接入步骤：

1. 继承 `MultiBrowserScraperBase`
2. 实现 `process_task()`
3. 按需要重写 `initialize_context()` 或 `initialize_page()`
4. 实例化时设置并发、Cookie、代理、超时参数
5. 用 `async with scraper:` 管理资源
6. 调用 `run_tasks(tasks)` 并发执行

如果只需要一个最小可用版本，业务层通常只实现 `process_task()` 就够了。

如果要做更复杂的站点适配，常见扩展点如下：

- `normalize_task()`: 做任务预处理
- `initialize_context()`: 注入 context 级配置
- `initialize_page()`: 注入 page 级初始化逻辑
- `classify_failure()`: 自定义失败归因
- `build_context_options()`: 定制 UA、语言、时区、代理等

## 6. 这个基类的优点

### 6.1 分层清晰

调度、执行、状态模型拆分得比较明确，业务代码不需要深入关心浏览器池和资源回收细节。

### 6.2 扩展点设计合理

通过 `process_task()`、`initialize_context()`、`initialize_page()`、`normalize_task()` 等钩子，业务层可以在不修改基类的前提下完成大多数站点适配。

### 6.3 并发模型实用

采用“浏览器复用 + context 隔离”的模型，兼顾了性能和任务隔离性，比“每任务一个浏览器”更轻，比“所有任务共享单 page”更稳。

### 6.4 任务级 Cookie 和代理能力较强

每个任务都可以带自己的 Cookie、代理和默认超时策略，适合需要多账号、多出口、多会话隔离的自动化场景。

### 6.5 异常处理有统一入口

失败分类集中在 `classify_failure()`，这让后续做统计、重试、回收、告警都更容易扩展。

### 6.6 资源回收考虑了并发安全

`_workers_recycling`、`_pool_condition` 和 `active_count` 的组合，解决了“正在回收的 worker 被再次分配”的典型竞态问题。

### 6.7 健康检查意识较强

`BrowserWorker.ensure_started()` 已经具备“浏览器引用存在但实际失效时重建”的能力，这对长期运行是有帮助的。

### 6.8 测试覆盖了关键回归点

当前测试已经覆盖了：

- 导入路径
- 浏览器健康检查
- 失败分类
- worker 回收期间不可重复获取
- 任务总超时后的状态更新

这说明基类已经开始进入“可维护、可回归验证”的阶段。

## 7. 这个基类的缺点与局限

### 7.1 `TaskPolicy` 已定义但尚未真正进入主流程

虽然目录里已经有 `TaskPolicy` 和 `CookieMode`，但当前 `MultiBrowserScraperBase` 仍然直接从 `task_data` 上取 `proxy`、`cookies` 等字段。

这意味着：

- 策略模型和实际执行链路还没有完全打通
- 任务字段仍偏松散
- 未来如果规则变复杂，可能会出现“字段越来越多但缺少统一收敛点”的问题

### 7.2 对第三方库私有能力有依赖

`BrowserWorker` 会访问或补丁化 Scrapling session 的私有成员，例如：

- `_context_options`
- `_initialize_context`
- `_config`
- 动态添加 `_build_context_with_proxy`

这类实现短期有效，但对第三方版本变更比较敏感，升级成本会偏高。

### 7.3 失败分类仍以关键字匹配为主

`classify_failure()` 目前的核心依据仍然是异常文本关键字，这种方式简单直接，但也存在几个问题：

- 新场景容易漏判
- 不同语言或不同依赖版本下的报错文本可能不一致
- 一条错误消息可能同时含有多种语义

它适合作为第一版通用分类器，但还不够“强语义”。

### 7.4 结果层没有内建重试机制

当前基类会做失败分类和 worker 状态记录，但不会自动根据分类执行重试。

这意味着：

- 遇到临时网络抖动时，需要业务层自己做重试
- `BAN_ERROR`、`PROXY_ERROR`、`UNKNOWN` 没有差异化重试闭环
- `TaskPolicy.max_retries` 暂时没有发挥作用

### 7.5 调度层没有更细的容量治理

现在的并发控制主要依赖：

- `browser_pool_size`
- `max_contexts_per_browser`

但缺少更进一步的控制手段，例如：

- 排队长度限制
- 任务优先级
- 分组隔离
- 单 worker 熔断
- 动态降载

所以它适合中小规模并发自动化，不一定适合极端高压或复杂多租户场景。

### 7.6 健康检查还不算完整

虽然已经有 `_is_browser_healthy()`，但它本质仍是轻量级检查，更多是在“任务即将执行前”判断 browser 是否失效。

目前还缺少：

- 独立看门狗
- 周期性巡检
- 更细粒度的内存/句柄健康指标
- 浏览器崩溃后的主动告警

对于长时间无人值守运行，这部分仍有提升空间。

### 7.7 观测能力偏弱

当前目录更偏“执行框架”，但还缺少系统性的观测字段，例如：

- 每个任务耗时
- 每个 worker 的成功率
- 每类失败的统计计数
- 浏览器重建次数
- 当前排队长度

没有这些信息时，排查慢性问题会更依赖人工日志分析。

### 7.8 文档与命名层还有提升空间

虽然源码注释已经比较完整，但目录内仍然缺少一份面向使用者的总览文档。对于第一次接入的人来说，理解成本主要来自：

- 不知道应该重写哪些方法
- 不清楚 worker 与 context 的关系
- 不清楚回收策略在哪一层生效

这也是本文档要补上的空白。

## 8. 适用场景与不适用场景

### 8.1 适用场景

这个基类比较适合：

- 需要多任务并发执行的 Playwright 自动化
- 多账号、多 Cookie 隔离的浏览器自动化
- 任务之间需要代理隔离的采集或页面操作
- 希望通过继承快速搭建业务爬虫/自动化脚本的场景

### 8.2 不太适合的场景

它当前不太适合：

- 需要非常复杂重试编排和租约调度的系统
- 需要分布式横向扩展的任务平台
- 需要极强观测、告警、熔断能力的生产级自动化平台
- 对第三方依赖版本稳定性要求极高、无法接受私有 API patch 的项目

## 9. 一句话总结

`driver_base` 的本质是一个“可继承的多浏览器自动化执行框架”。它最大的价值在于把并发调度、浏览器复用、任务隔离、失败分类和资源回收这些共性能力统一收口，让业务层只关注页面逻辑；它当前的主要短板则是策略模型尚未完全接入、对第三方私有 API 有依赖，以及生产级观测和重试能力还不够完整。

## 10. 后续建议

如果后续准备继续演进这个基类，建议优先级可以按下面排序：

1. 把 `TaskPolicy` 正式接入执行主链路
2. 给失败分类和重试机制打通闭环
3. 把第三方私有 API 访问收敛到单独适配层
4. 增加任务/worker 级监控指标
5. 补充长期运行场景下的看门狗与健康巡检
