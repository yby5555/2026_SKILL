## 2. 关键问题分析

### 🔴 高优先级问题

#### 2.1 浏览器崩溃后缺少自动重建机制

**位置**: `browser_worker/__init__.py:165-177`

```python
async def close(self) -> None:
    if self._session is not None:
        await self._session.close()
    self._session = None
    self._browser = None  # ← 只是置空，不会自动重建
```

**问题描述**:
- 当浏览器崩溃（如 Chrome GPU 进程崩溃、内存溢出、段错误）时
- `BrowserWorker` 被标记为 `request_recycle = True`
- 但 **worker 池中没有自动重建 worker 的机制**
- 一旦 worker 被关闭，它将永久失效

**风险**: 长期运行时，可用 worker 会越来越少，最终导致整个爬虫停止工作。

---

#### 2.2 健康检查缺失

**位置**: `browser_worker/__init__.py:122-127`

```python
async def ensure_started(self) -> None:
    if self._browser is not None:
        return  # ← 仅检查引用是否为空，不检查浏览器是否存活
```

**问题描述**:
- 只在首次启动时检查浏览器是否存在
- **没有定期验证浏览器进程是否仍然存活**
- 浏览器可能已崩溃（`SIGABRT`、GPU 进程退出），但 `_browser` 对象仍然非空

**风险**: 任务会被分配到已死亡的浏览器，导致任务失败。

**可能的崩溃信号**:
- `Target closed`
- `Session not found`
- `CDP disconnected`
- WebSocket 连接断开（Playwright 通过 CDP 与浏览器通信）

---

### 🟡 中优先级问题

#### 2.3 内存泄漏风险点

**位置**: `browser_worker/__init__.py:270-280`

```python
finally:
    if page is not None:
        try:
            await page.close()
        except Exception:
            pass
    if context is not None:
        try:
            await context.close()
        except Exception:
            pass
```

**问题描述**:
- 虽然有 `page.close()` 和 `context.close()`，但：
  - **没有显式清理 JavaScript 句柄（JSHandle）**
  - **没有清理大型 DOM 引用（ElementHandle）**
  - **Context 可能持有缓存资源**

**风险**: Chrome DevTools Protocol 的资源如果不显式释放，会累积导致内存增长，最终触发 OOM。

---

#### 2.4 失败分类可能遗漏崩溃场景

**位置**: `multi_browser_scraper_base/__init__.py:130-143`

```python
_WORKER_ERROR_HINTS = (

    "browser has been closed",

    "page has been closed",

```

**遗漏的场景**:
| 场景 | 典型错误信息 |
|------|--------------|
| 系统信号崩溃 | `SIGSEGV`, `SIGABRT` |
| 静默崩溃 | 浏览器进程已退出但 Python 对象仍存在 |
| CDP 断连 | `WebSocket closed`, `CDP session disconnected` |
| 句柄失效 | `TargetID mismatch`, `Invalid target` |

---

#### 2.5 Worker 回收期间可能导致任务积压

**位置**: `multi_browser_scraper_base/__init__.py:734-748`

```python
if current_active == 0 and worker.request_recycle:
    self._workers_recycling.add(worker.worker_id)  # 标记为回收中
    should_recycle = True
```

**问题描述**:
- 如果回收耗时较长（关闭浏览器需要几秒），期间该 worker 不可用
- **没有队列大小限制**
- **等待任务会持续占用内存**
- **可能导致级联超时**

**风险**: 在高负载下，多个 worker 同时回收会导致任务大量积压。

---

### 🟢 低优先级问题

#### 2.6 重试策略未完全打通

**位置**: `task_policy/__init__.py:11-39`

`TaskPolicy` 已定义但 **在 `MultiBrowserScraperBase` 中未被使用**：
- 没有按 `FailureCategory` 区分重试策略
- 代理错误不应该触发浏览器回收，但应该触发代理切换
- `BAN_ERROR` 应该暂停而非立即重试

---

#### 2.7 依赖第三方私有 API

**位置**: `browser_worker/__init__.py:139-161`

```python

    self._session._config, context

```

**风险**: Scrapling 库升级时可能破坏这些私有访问。

## 3. 优化建议汇总

| 优先级 | 问题 | 建议方向 | 预期收益 |
|--------|------|----------|----------|
| **高** | 浏览器崩溃后无法自动重建 | 增加 worker 健康检查 + 自动重启机制 | 避免 worker 池逐渐萎缩 |
| **高** | 缺少浏览器存活验证 | 定期 ping CDP 或执行轻量操作验证 | 避免任务分配到已死浏览器 |
| **中** | 内存泄漏风险 | 显式清理 JS 句柄、添加内存监控 | 延长单次运行周期 |
| **中** | 崩溃关键字遗漏 | 增加异常类型判断（如 `CDPDisconnected`） | 提高崩溃检测准确率 |
| **中** | 回收期间任务积压 | 限制等待队列、支持快速失败 | 避免级联超时 |
| **低** | 重试策略未打通 | 将 `TaskPolicy` 集成到执行流程 | 更精细的错误处理 |
| **低** | 依赖私有 API | 封装适配层隔离第三方变化 | 提高版本兼容性 |

## 4. 核心优化方案

### 4.1 浏览器健康检查机制

**建议在 `BrowserWorker.ensure_started()` 中增加**:

```python
async def ensure_started(self) -> None:
    """确保 worker 底层浏览器已经启动且健康。"""
    if self._browser is not None:
        # 新增：检查浏览器是否仍然存活
        if not await self._is_browser_alive():
            await self.close()  # 清理已死的浏览器
        else:
            return

    async with self._start_lock:
        # ... 原有启动逻辑 ...

async def _is_browser_alive(self) -> bool:
    """检查浏览器是否仍然存活。"""
    if self._browser is None or self._session is None:
        return False

    try:
        # 轻量级检查：获取浏览器版本
        version = await self._browser.version()
        return bool(version)
    except Exception:
        return False
```

### 4.2 自动重建机制

**建议在 `MultiBrowserScraperBase` 中增加**:

```python
async def _acquire_worker(self) -> BrowserWorker:
    """获取一个当前可用的 worker，失败时自动重建。"""
    await self.start()

    while True:
        async with self._pool_condition:
            # 过滤出健康的 worker
            available = [
                worker
                for worker in self._workers
                if self._worker_active_counts.get(worker.worker_id, 0) < worker.max_contexts
                and worker.worker_id not in self._workers_recycling
            ]

            if available:
                worker = min(available, key=lambda item: (...))
                self._worker_active_counts[worker.worker_id] += 1

                # 新增：使用前检查健康状态
                try:
                    await worker.ensure_started()
                except Exception:
                    # worker 已损坏，标记回收并继续选择
                    worker.request_recycle = True
                    self._workers_recycling.add(worker.worker_id)
                    continue

                return worker

            await self._pool_condition.wait()
```

**建议在 `run_task()` 的 finally 块中增加**:

finally:

### 4.5 看门狗机制（可选）

**建议在 `MultiBrowserScraperBase` 中增加**:

```python
async def _watchdog_loop(self, interval_seconds: int = 30):
    """定期检查所有 worker 的健康状态。"""
    while not self._closed:
        await asyncio.sleep(interval_seconds)

        for worker in self._workers:
            if worker._browser is not None:
                try:
                    # 发送轻量级心跳
                    await worker._browser.version()
                except Exception:
                    # 标记需要回收
                    worker.request_recycle = True
```
