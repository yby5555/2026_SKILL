# VEO 任务推送队列说明

> 适用对象：VEO 任务消费者、队列维护人员、失败重试逻辑开发人员

## 1. 队列用途

VEO 任务接口在调用方成功创建任务后，不会直接执行生成逻辑，而是会将任务推送到 Redis 队列，由消费者异步拉取并处理。

整体链路如下：

1. 调用方请求 `POST /internal_api/daas/veo/task/create`
2. 服务端完成参数校验
3. 服务端将任务写入 Mongo 任务表
4. 服务端将任务推入 Redis 队列
5. 消费者从 Redis 队列取出任务并执行
6. 消费者执行完成后回写任务状态

---

## 2. 队列类型

当前推送队列使用的是 Redis `zset`，也就是有序集合。

服务端入队时使用的是 `zadd`，因此从实现类型上，它属于优先级队列，而不是普通列表队列。

### 当前特点

- 队列类型：优先级队列
- Redis 结构：`zset`
- 当前生产者默认优先级：`10`
- 实际写入 Redis 的 `score`：由“优先级 + 创建时间”共同编码生成
- 队列 key 配置项：`redis.priority_queue_key`
- 默认队列 key：`task:create:queue`

### 说明

- 当前生产者默认优先级仍然是 `10`
- 服务端会把“优先级”和“创建时间”一起编码进 `score`
- 因此同优先级任务在按最大 `score` 消费时，会按先入先出处理
- 后续如果消费者需要将失败任务重新推回 Redis，只需要写入同一个 `zset`，并为该任务设置更高的分值即可

---

## 3. 优先级规则

当前约定如下：

- 数值越大，优先级越高
- 普通新建任务默认优先级为 `10`
- 实际 `score` 由“优先级主排序 + 时间戳次排序”组成
- 同优先级下，越早创建的任务 `score` 越大
- 如果后续失败重试任务需要优先执行，可以使用大于 `10` 的优先级，例如 `20`、`50`、`100`

### 重要提醒

如果要真正按“高优先级先执行”，消费者取任务时必须按“取最大 score”的方式消费。

也就是说，消费者侧应采用以下思路之一：

- 使用 `ZPOPMAX`
- 使用 `BZPOPMAX`
- 或先按倒序读取，再删除对应成员

如果消费者按最小 score 取数据，那么就会和当前“数值越大越优先”的规则相反。

### 同优先级顺序

当前服务端已经按下面规则编码 `score`：

- 第一排序位：任务优先级
- 第二排序位：创建时间

因此在消费端按最大 `score` 出队时，顺序表现为：

1. 高优先级任务优先
2. 同优先级任务按先入先出

说明：

- 这里的“先入先出”依赖的是服务端写入 `score` 时编码的创建时间
- 如果消费者重推任务时要保留同优先级 FIFO 语义，建议也沿用同样的编码规则

---

## 4. 推送消息格式

服务端推入 Redis 的成员内容是 JSON 字符串，业务字段如下。

### 4.1 无图片任务

```json
{
  "_id": "b5820183-0958-45e9-8b59-d9919b769bc8",
  "prompt": "生成一个电影感的短视频，城市夜景，镜头缓慢推进",
  "type": 1
}
```

### 4.2 有图片任务

```json
{
  "_id": "b5820183-0958-45e9-8b59-d9919b769bc8",
  "prompt": "生成一个电影感的短视频，城市夜景，镜头缓慢推进",
  "type": 1,
  "image_value": ["xxx"],
  "image_type": "base64"
}
```

### 4.3 字段说明

| 字段 | 类型 | 是否必有 | 说明 |
| --- | --- | --- | --- |
| `_id` | string | 是 | 任务唯一 ID |
| `prompt` | string | 是 | 任务提示词 |
| `type` | int | 是 | 任务类型，`0`=图片，`1`=视频 |
| `image_value` | array | 否 | 参考图列表；当前生产者默认放 1 个值 |
| `image_type` | string | 否 | 图片类型，当前支持 `url` 或 `base64` |

### 4.4 图片字段规则

- 当请求中未传图片时，不会推送 `image_value`
- 当请求中未传图片时，不会推送 `image_type`
- 当图片是 URL 时，`image_type = "url"`
- 当图片是 Base64 时，`image_type = "base64"`

---

## 5. 推荐的消费者处理方式

消费者拿到任务后，建议按以下顺序处理：

1. 解析 Redis 成员中的 JSON 字符串
2. 读取 `_id`、`prompt`、`type`
3. 判断是否存在 `image_value` 和 `image_type`
4. 执行业务处理
5. 成功时回写任务完成状态
6. 失败时记录失败原因，并按需要决定是否重推队列

---

## 6. 失败重推建议

如果消费者执行失败，需要重新推回队列，建议遵循以下规则：

- 重推时继续使用同一个 Redis 队列 key
- 重推消息体字段结构保持不变
- 重推任务的优先级设置为更高值，以便优先执行
- 重推时也应重新按“优先级 + 当前时间”生成新的 `score`
- 建议在消费者侧额外记录重试次数，避免死循环重试

示例：

- 首次任务：priority = `10`
- 一次失败重推：priority = `20`
- 紧急人工重推：priority = `100`

---

## 7. 当前结论

当前 VEO 任务推送队列已经是优先级队列，原因如下：

- 底层使用 Redis `zset`
- 服务端使用 `zadd` 入队
- 队列使用 `score` 承载“优先级 + 时间顺序”
- 后续可以直接通过调整 score 支持失败任务优先重试

当前唯一需要消费者配合确认的是：

- 消费端必须按“最大 score 优先”的方式取任务

只要消费端遵守这一点，这套队列就可以完整支持后续的优先级重推方案。
